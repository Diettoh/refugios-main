**Section 2 — Replace source CHECK constraint (RES-04):**
This is the most critical section. Follow the exact order:
1. Use a DO block with `pg_constraint` catalog to discover and DROP the existing source CHECK constraint by name (do not hardcode the constraint name — it may be auto-generated differently in dev vs prod).
2. UPDATE existing rows to new vocabulary:
   - `UPDATE reservations SET source = 'direct' WHERE source IN ('phone', 'walkin');`
3. ADD new constraint with explicit name: `ALTER TABLE reservations ADD CONSTRAINT reservations_source_new_check CHECK (source IN ('booking', 'airbnb', 'web', 'direct', 'other'));`

Wrap all three sub-steps in a single DO block so they run atomically. Make it idempotent: before dropping, check if the new constraint already exists (if it does, skip the entire section).

```sql
DO $$
DECLARE v_old_constraint TEXT;
BEGIN
  -- Skip if already migrated (new constraint exists)
  IF EXISTS (
    SELECT 1 FROM pg_constraint
    WHERE conrelid = 'reservations'::regclass
      AND conname = 'reservations_source_new_check'
  ) THEN
    RETURN;
  END IF;

  -- Drop old constraint
  SELECT conname INTO v_old_constraint
  FROM pg_constraint
  WHERE conrelid = 'reservations'::regclass
    AND contype = 'c'
    AND pg_get_constraintdef(oid) LIKE '%source%';
  IF v_old_constraint IS NOT NULL THEN
    EXECUTE 'ALTER TABLE reservations DROP CONSTRAINT ' || quote_ident(v_old_constraint);
  END IF;

  -- Backfill to new vocabulary
  UPDATE reservations SET source = 'direct' WHERE source IN ('phone', 'walkin');

  -- Add new constraint
  ALTER TABLE reservations
    ADD CONSTRAINT reservations_source_new_check
    CHECK (source IN ('booking', 'airbnb', 'web', 'direct', 'other'));
END $$;
```
